# Hybrid Clade Data Strategy (YAML + SQLite)

## Recommendation

Use a hybrid data model:

1. **OpenTree of Life API** for taxonomy/topology (IDs, lineage, subtree shape).
2. **Paleobiology Database (PBDB) API** (or curated fossil refs) for temporal ranges.
3. **YAML** as curated, app-facing backbone for broad/mid-level clades.
4. **SQLite** for deeper descendant detail under selected handoff clades.

This keeps the UI fast and deterministic while allowing much larger clade depth.

## Why This Source Stack

### OpenTree (topology)
- Strong for: name resolution (TNRS), lineage checks, subtree relationships.
- Weak for: geological age ranges suitable for timeline display.

### PBDB / curated fossil source (time)
- Stronger for first appearance / range estimates.
- Needed to populate `start_ma`, `end_ma`, confidence, and notes.

## Architecture

## 1) YAML Core (small, curated)

Keep `data/clades.yaml` as the primary display backbone:
- broad and mid-level clades
- stable labels and UI metadata
- zoom/display priority fields

Add explicit handoff metadata on clades that should load deep descendants from SQLite:

```yaml
- id: dinosauria
  scientific_label: Dinosauria
  common_label: Dinosaurs
  zoomable: true
  detail_source: sqlite
  detail_scope: descendants
```

Do **not** use numeric ID thresholds (for example, `>5000`) as a rule.  
Use explicit handoff markers per clade.

## 2) SQLite Detail Layer (large, generated)

Store deep descendants in SQLite tables, generated by script.

Suggested minimum schema:

```sql
clades_detail(
  id TEXT PRIMARY KEY,
  parent_id TEXT,
  ott_id INTEGER,
  opentree_name TEXT,
  scientific_label TEXT NOT NULL,
  common_label TEXT,
  scientific_rank TEXT,
  start_ma REAL,
  end_ma REAL,
  divergence_ma REAL,
  range_note TEXT,
  confidence TEXT,
  uncertainty TEXT,
  short_description TEXT,
  extinction_note TEXT,
  display_groups_json TEXT,
  display_priority INTEGER,
  min_zoom_level TEXT,
  representative_taxa_json TEXT,
  tags_json TEXT,
  branch_priority INTEGER,
  cladistic_role TEXT,
  zoomable INTEGER NOT NULL DEFAULT 0,
  include_in_main_tree INTEGER,
  collapsed_by_default INTEGER,
  opentree_json TEXT,
  age_confidence TEXT,
  source_topology TEXT,
  source_age TEXT,
  updated_at TEXT
);

clade_detail_roots(
  root_id TEXT NOT NULL,
  descendant_id TEXT NOT NULL,
  depth INTEGER,
  PRIMARY KEY(root_id, descendant_id)
);
```

Notes:
- `display_groups_json`, `representative_taxa_json`, and `tags_json` preserve current YAML list fields.
- `opentree_json` mirrors optional nested `opentree` metadata when present.
- Boolean fields use `INTEGER` (`0/1`) for SQLite portability.

## 3) Runtime Behavior

1. App loads YAML core at startup (offline, deterministic).
2. User zooms into a `detail_source: sqlite` clade.
3. App loads only that subtree from SQLite.
4. Renderer merges YAML root node + SQLite descendants by canonical IDs.

## Data Pipeline

Create/extend a build script (for example `scripts/build_clade_hybrid_data.py`) to:

1. Resolve clade names with OpenTree TNRS.
2. Pull OpenTree lineage/subtree for selected roots.
3. Pull age signals from PBDB/curated source.
4. Apply reconciliation rules (OpenTree topology + age source).
5. Emit:
   - updated curated YAML core
   - SQLite detail DB
   - report in `docs/` with mismatches and missing ages

## Reconciliation Rules

1. **Topology authority**: OpenTree (unless manually overridden).
2. **Age authority**: PBDB/curated fossil source.
3. If age missing:
   - keep clade in detail DB with null age fields
   - mark `age_confidence: missing`
   - include in report
4. Never auto-delete curated YAML nodes without explicit review.

## Versioning and Maintenance

1. Add `clade_data_version` metadata table in SQLite.
2. Include pipeline run date and source versions in report.
3. Treat SQLite as generated artifact; do not hand-edit it.
4. Keep human edits in curated YAML or override files only.

## Rollout Plan

### Stage 1
- Keep current YAML-only runtime.
- Build script produces SQLite detail DB for one root (`dinosauria`) and validation report.

### Stage 2
- Enable runtime SQLite subtree loading only when `detail_source: sqlite`.
- Keep fallback to YAML-only if DB missing.

### Stage 3
- Add additional roots (`mammalia`, `arthropoda`, etc.) using same handoff model.

## Risks

1. Taxonomy drift between sources.
2. Missing or conflicting age estimates.
3. ID instability if not anchored to canonical IDs (OTT where possible).

Mitigation:
- strict validation report
- deterministic reconciliation rules
- curated override layer

## Conclusion

This hybrid model is a good long-term fit:
- YAML remains clean and maintainable for visible structure.
- SQLite scales for deep clade detail.
- OpenTree + PBDB gives strong topology + time coverage without runtime API dependency.
